## Analysis
**FIXED 2026-06-22 (wt-soundness-loop-4) via a scoped `Base.hvcat` overlay.** The tuple-element
matrix literal `[(r,g,b,a) (r,g,b,a); …]` lowers to `Base.hvcat((nc,nc,…), tup, tup, …)` (confirmed
via `Meta.lower`), which routes through `Base._typed_hvncat_dims` — WT stubs that 1000+-stmt
machinery to `unreachable` → runtime trap. The Triage below notes the known-working alternative
(`Matrix{T}(undef, m, n)` + element stores) and that *whitelisting* the Base helpers StackOverflows
discovery (recursive NTuple-element-matrix type registration). FIX: overlay `Base.hvcat` scoped to
`values::T... where T<:Tuple` that reconstructs the rectangular matrix via the working undef+stores
path — sidesteps both `_typed_hvncat` AND the StackOverflow. Numeric hvcat (`values::Number`) is a
different dispatch so it's untouched. Ragged row specs (which native rejects) throw here too →
parity, never a wrong shape (soundness). VERIFIED: gap repro returns 1 (native=1) bit-exact; a 2×3
tuple matrix returns 15; numeric `[1.0 2.0;3.0 4.0]` / `[1 2 3;4 5 6]` still match native; full
Pkg.test green; loop_guard clean. Real WasmMakie relevance: this is exactly `image!` RGBA data.

## Triage (P3 bugsmash, 2026-06-12)

Plain-element matrix literals ([1.0 2.0; 3.0 4.0]) work after the
matrix-layout fix (gap 3aaa51b9a688). The TUPLE-element shape routes
through Base._typed_hvncat_dims/_typed_hvncat_shape (1049-stmt
machinery, concrete rettype) which currently stubs → runtime trap.
Whitelisting those helpers was attempted and REVERTED: compiling them
sends the discovery/compile pipeline into a StackOverflow (recursive
type registration or discovery loop on the NTuple-element matrix
types) — a compiler crash, strictly worse than the trap. Needs the
recursion diagnosed (stacktrace via ulimit/smaller depth) before the
whitelist can land. Workaround for consumers: build the matrix with
Matrix{NTuple{4,Float64}}(undef, m, n) + explicit stores (works).
