## Postmortem (FIXED)

Root cause: the 4th instance of the backward byte-scan misparse class.
The "dead return at end of body" peephole in `generate_body`
(generate.jl) pattern-matched the last four RAW BYTES of the function
body: `[end][return][unreachable][end]`. On 1.13 this function's IR
allocates more locals, so the return value lands in local **1418**,
whose LEB128 encoding is `0x8a 0x0b` — the trailing `0x0b` byte mimics
an END opcode. The genuine tail `[local.get 1418][return]
[unreachable][end]` therefore matched the dead-return pattern and the
peephole rewrote the LIVE `return` to `unreachable`. The old guard only
inspected ONE byte before the suspected END (looking for an
operand-taking opcode), but here that byte is the LEB continuation
byte `0x8a`, not the `local.get` opcode two bytes back.

1.12 passed only by accident: its IR has fewer locals, the value lands
in local 1302 (`0x96 0x0a`), and `0x0a` ≠ END. Any function whose
returned local index lies in 1408..1535 (second LEB byte `0x0b`) would
trip this on either version.

Fix: forward-parse the body with the `_last_instr_starts(bytes, 4)`
helper (refactored from `_last_instr_start` into a shared
`_instr_next` single-instruction stepper, now also covering
`try_table`, `throw`, and `br_on_null/non_null` immediates) and match
the four genuine final INSTRUCTIONS instead of raw bytes. Both
dead-return patterns are preserved; immediate bytes can no longer
masquerade as opcodes.

Prior instances of this class: i32.const-32-as-local.get (titlecase),
call-LEB trailing scan, local.get-379-as-struct_get (a6c6091b2a80).
The lesson stays the same: NEVER pattern-match wasm bytes backwards —
always forward-parse from a known instruction boundary.
