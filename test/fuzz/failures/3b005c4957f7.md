## Analysis
**Root cause: `length(::String)` (Base's UTF-8 char-count loop) traps as a
*map-kernel dependency*.** `map(y->length(s), v)` traps for any non-empty `v`,
while map-safe field reads (`ncodeunits`/`sizeof`) are fine — same class as the
now-fixed `asin`/Base-`length` map-kernel traps.

**Why this is NOT overlay-fixable (a trialed overlay was reverted).** The obvious
fix is a `length(::String)` overlay that lead-byte-counts via the map-safe
`ncodeunits`/`codeunit` primitives. It *works* in map and standalone — but it
**silently breaks the hot strip path**:

- `lstrip(::String)`/`rstrip(::String)` (interpreter.jl) call `length(s)`
  internally and use it as a **byte** count (`codeunit(s, 1:n)`), relying on
  char==byte for ASCII — a deliberate workaround for the `ncodeunits`-on-
  `String(bytes)` aliasing bug (see `0beb5ec969a2`).
- The overlay's `(codeunit(s,i) & 0xc0) != 0x80 && (c += 1)` body miscompiles
  **in that dependency context** — the same boolean-cond-in-`while` codegen class
  the strip overlays already contort around — so the previously-green **ASCII**
  strip/lstrip/rstrip tests (and STRESS-3001 string chains) fail.
- Confirmed by full `Pkg.test()`: the overlay turned 0 → 19 failures, all
  strip/`length`-composed; reverting it restores green. (The no-strip chain
  `chain3` stayed green throughout — isolating strip as the differentiator.)

A global `length(::String)` that is correct **both** as a map kernel **and**
inside `lstrip`/`rstrip` requires the underlying map-kernel-dependency codegen bug
(and the `ncodeunits`-on-`String(bytes)` aliasing bug) fixed first — then both this
gap and the strip-unicode gap (`0beb5ec969a2`) close together. **Deferred to
Part 2** (deep codegen, not overlay-fixable without breaking strip).
