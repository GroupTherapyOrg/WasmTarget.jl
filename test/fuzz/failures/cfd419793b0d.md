## Analysis
**Reframed (2026-06-13, Pluto campaign).** `string(::Float64)`/Ryu now WORKS standalone
AND in module-context (the old `19d59e9a61b3` unbalanced-control-frames defect is gone —
verified `string(1.5)`, `string(12345.678)` both with and without bridge accessors). So this
gap is NOT downstream of Ryu anymore. `string(::Complex)` was hiding **two** distinct
codegen bugs:

1. **structref-vs-concrete-ref type-precision (FIXED, this campaign).** Base's
   `show(io, z::Complex)`/`print` specialize on an abstract value arg, so the wasm param
   is declared `structref` (`get_concrete_wasm_type` types `<: Number` structs abstract —
   `is_struct_type` returns false for `T <: Number`). The body then does `struct.get
   $Complex` / nested `struct.new` against that `structref` → `type mismatch: expected
   (ref null $type), found structref`. Fixed by extending `emit_ref_cast_if_structref!`
   (context.jl) to the `Core.Argument` (param) case AND adding the analogous abstract→
   concrete `ref.cast` on the struct.new field path (statements.jl, the ConcreteRef
   branch). Same root closes **RGB{N0f8}** bridge ctors (PI fractals/conv2d/images) and
   any numeric-struct (Rational/Complex) ctor/field access. Guarded by
   `@testset "Numeric-struct field struct.new/struct.get"` (Base-only Complex{Rational}).

2. **show-machinery dead-value — ADDRESSED via overlay (2026-06-14).** With (1) fixed,
   the failure moved to `func $nonnothing_nonmissing_typeinfo` (Base typeinfo/show helper):
   `nonmissingtype(nonnothingtype(get(io,:typeinfo,Any)))` does RUNTIME type subtraction
   over the type lattice, which the backend can't lower — it stubbed to `unreachable` and
   the surrounding `block` underflowed (`ref.is_null` on an empty stack; the dead-value/
   stackifier class shared with median/Printf-`format`). FIXED with an `@overlay` in
   interpreter.jl: `nonnothing_nonmissing_typeinfo(io::IO) = Any`. For the plain-IOBuffer
   path that float/Complex formatting uses, `get(io,:typeinfo,Any) === Any` and
   `nonmissingtype(nonnothingtype(Any)) === Any`, so this is EXACT; inference const-folds
   it, sidestepping the stackifier defect. (Only typeinfo-CONTEXT container display would
   differ — oracle-gated in PlutoIslands.)

**NET: `string(::Complex)` now emits a VALID, CORRECT module** (verified: compiles +
wasm-tools-validates; reaches `WebAssembly.instantiate`). The fuzz harness can't RUN it —
the module imports the `wasm:js-string` builtin which the harness's importObject doesn't
provide (a harness limitation; PlutoIslands' runtime DOES provide it). So the CODEGEN gap
is closed; this gap stays `open` only because the bare fuzz harness can't exec-verify it
(blocked on adding js-string to the harness importObject — a separate harness task).
Guarded by `@testset "string(::Complex) compiles (typeinfo overlay)"` (validate-only).

### UPDATE 2026-06-18 (IOBuffer-memmove deep dive — STILL OPEN, multi-layer)

Verified via `FuzzBridge.bridge_run` (js-string enabled) that this is a **runtime
trap**, not a harness js-string false-open. Drilled the full `string(::Complex)` →
`show(io::IOBuffer, ::Complex)` path and found it sits on the **general Base IOBuffer
string-building machinery, which WT deliberately does NOT support** (it overlays
specific string fns instead — see `interpreter.jl:80-82,500`; the float path works via
a `StringVector` overlay, NOT a plain IOBuffer). Layers found, outermost-first:

1. **memmove/pointerref string-IO stubs (FIXABLE — patch derived + saved).** WT's
   memmove/pointerref tracers couldn't follow 4 patterns, so each no-op'd
   (type-default) or stubbed `unreachable`, corrupting the buffer:
   - **A** `_trace_memmove_ptr` getfield-terminal rejected `getfield(io,:data)::Memory{UInt8}`
     (intra-array ensureroom compaction). Fix: accept Memory/GenericMemory terminal.
   - **B** memmove src = `jl_string_to_genericmemory(::String)`. Fix: terminal returns the
     String (String/Vector{UInt8}/Memory{UInt8} share the packed-i8 array type, so the
     String value IS the array; `_emit_backing_array!` extended with `vt===String`).
   - **C** memmove src = `jl_string_ptr(STRING CONST)` — **base-1** (jl_string_ptr→`i64.const 1`,
     not 0), so src offset = (ptr value − 1); dest (io.data) stays base-0. Dedicated
     memmove block + `_trace_string_ptr_terminal` helper. ⚠️ This −1 is UNVERIFIED
     end-to-end (see below) — silent-corruption risk.
   - **D** `pointerref(BARE jl_string_ptr(const), i, _)` — `_trace_string_ptr` only handles
     add_ptr/sub_ptr (`:call`) wrappers, missing the bare `:foreigncall`. Fix: bare-case
     in the pointerref handler reads `str[i-1]`.
   After A-D ALL memmove `MMtrace FAIL` + pointerref `UNREACH` are gone. Patch saved at
   `~/.claude/projects/.../memory/wt-iobuffer-memmove-AD.patch` (loop_guard: clean).
2. **empty-IOBuffer → NULL data array (DEEPER, pre-existing, BLOCKER).** With A-D applied,
   the trap MOVES to `array.len` on an explicit `ref.null` (i8-array). Minimal repro
   `wonly(x)=(io=IOBuffer(); write(io,"hello world"); String(take!(io)))` traps "null
   pointer"; `wpre(x)=String(take!(IOBuffer("hi")))` (pre-filled) WORKS. So **empty
   `IOBuffer()`/`IOBuffer(sizehint=n)` constructs a null `.data` Memory**; `ensureroom`'s
   `array.len(io.data)` then derefs null. Distinct from A-D (no WT edit emits ref.null).
3. **further layers**: `sprint(print, s)` → `unreachable`; varargs `string(a,x,b)` →
   3-arg `getfield(_2,i,boundscheck)` (tuple iteration) `unreachable`.

**VERDICT: this is a dedicated multi-session project (implement, or overlay, the whole
Base IOBuffer string-building path), NOT a loop-paced fix.** A-D are real + necessary but
NOT sufficient, and their array.copy OFFSETS can't be differential-verified until layer 2
is fixed (no write+take! case yet yields output) — so per the loop's verify-before-commit
guardrail they were NOT committed (tree reverted clean). **Recommended next step: a
Complex/IOBuffer-show OVERLAY (mirroring the float-string overlays in interpreter.jl),
which is WT's established design for unsupported Base string machinery — rather than
implementing Base IOBuffer construction+growth wholesale.** Re-apply the A-D patch only
once layer 2 unblocks end-to-end differential verification of the offsets.

### RESOLVED 2026-06-18 (overlay) — gap CLOSED

Took the overlay route (Dale's call). Added `@overlay WASM_METHOD_TABLE
Base.string(z::Complex)` in `interpreter.jl` (beside the `*`/string overlays): it
byte-assembles the result — `string(real)` + `" + "`/`" - "` + `string(|imag|)` +
optional `"*"` + `"im"` — mirroring `Base.show(io, ::Complex)` EXACTLY (sign via
`signbit(i) && !isnan(i)`; the `"*"` separator when imag is non-finite/non-Integer),
reusing `string(::Real)` (which WT supports via the Ryu/StringVector overlays) and only
`ncodeunits`/`codeunit`/`push!`/`String(bytes)`. This bypasses the entire unsupported
Base IOBuffer path (L1-L3 above), so NONE of the A-D memmove work was needed for THIS gap.

Differential-verified byte-exact vs native via `FuzzBridge.bridge_run` across 24 cases
(the gap value; negative imag `" - "`; `Inf`/`NaN` → `"*im"`; integer-valued `2.0+3.0im`;
zero; over x∈{0,1,3,7}) — zero mismatches. Gap auto-closes via the updated bridge
reproducer (`bridge_run_args(...; rettype=String)` + `tree_matches`); the old
`compile_and_run_vec` reproducer was a permanent false-OPEN for String returns (it
JSON.stringifies the array ref to `"undefined"` AND lacks js-string). Regression guard:
`@testset "string(::Complex) runs byte-exact (overlay)"` in runtests.jl (scalar len/codeunit
checks, exec without js-string).

NOTE: the A-D memmove/pointerref tracer work + the L2 null-empty-IOBuffer bug remain a
separate, still-OPEN general-IOBuffer codegen project (patch: `memory/wt-iobuffer-memmove
-AD.patch`) — valuable for `take!`/`write`/`sprint`/varargs-`string` broadly, but NOT
required for `string(::Complex)`/`::Rational`/etc., which the overlay now handles.
