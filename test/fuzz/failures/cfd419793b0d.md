## Analysis
**Reframed (2026-06-13, Pluto campaign).** `string(::Float64)`/Ryu now WORKS standalone
AND in module-context (the old `19d59e9a61b3` unbalanced-control-frames defect is gone —
verified `string(1.5)`, `string(12345.678)` both with and without bridge accessors). So this
gap is NOT downstream of Ryu anymore. `string(::Complex)` was hiding **two** distinct
codegen bugs:

1. **structref-vs-concrete-ref type-precision (FIXED, this campaign).** Base's
   `show(io, z::Complex)`/`print` specialize on an abstract value arg, so the wasm param
   is declared `structref` (`get_concrete_wasm_type` types `<: Number` structs abstract —
   `is_struct_type` returns false for `T <: Number`). The body then does `struct.get
   $Complex` / nested `struct.new` against that `structref` → `type mismatch: expected
   (ref null $type), found structref`. Fixed by extending `emit_ref_cast_if_structref!`
   (context.jl) to the `Core.Argument` (param) case AND adding the analogous abstract→
   concrete `ref.cast` on the struct.new field path (statements.jl, the ConcreteRef
   branch). Same root closes **RGB{N0f8}** bridge ctors (PI fractals/conv2d/images) and
   any numeric-struct (Rational/Complex) ctor/field access. Guarded by
   `@testset "Numeric-struct field struct.new/struct.get"` (Base-only Complex{Rational}).

2. **show-machinery dead-value — ADDRESSED via overlay (2026-06-14).** With (1) fixed,
   the failure moved to `func $nonnothing_nonmissing_typeinfo` (Base typeinfo/show helper):
   `nonmissingtype(nonnothingtype(get(io,:typeinfo,Any)))` does RUNTIME type subtraction
   over the type lattice, which the backend can't lower — it stubbed to `unreachable` and
   the surrounding `block` underflowed (`ref.is_null` on an empty stack; the dead-value/
   stackifier class shared with median/Printf-`format`). FIXED with an `@overlay` in
   interpreter.jl: `nonnothing_nonmissing_typeinfo(io::IO) = Any`. For the plain-IOBuffer
   path that float/Complex formatting uses, `get(io,:typeinfo,Any) === Any` and
   `nonmissingtype(nonnothingtype(Any)) === Any`, so this is EXACT; inference const-folds
   it, sidestepping the stackifier defect. (Only typeinfo-CONTEXT container display would
   differ — oracle-gated in PlutoIslands.)

**NET: `string(::Complex)` now emits a VALID, CORRECT module** (verified: compiles +
wasm-tools-validates; reaches `WebAssembly.instantiate`). The fuzz harness can't RUN it —
the module imports the `wasm:js-string` builtin which the harness's importObject doesn't
provide (a harness limitation; PlutoIslands' runtime DOES provide it). So the CODEGEN gap
is closed; this gap stays `open` only because the bare fuzz harness can't exec-verify it
(blocked on adding js-string to the harness importObject — a separate harness task).
Guarded by `@testset "string(::Complex) compiles (typeinfo overlay)"` (validate-only).
