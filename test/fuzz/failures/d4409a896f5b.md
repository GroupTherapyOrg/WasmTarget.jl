## Analysis
**FIXED 2026-06-22 (wt-dyndispatch).** The `try/finally` + `daysinmonth` shape was a red
herring. Minimized (try/finally with a value, `div`, etc. all compile fine): the trap is
`daysinmonth(y, m)`, whose Base body does `DAYSINMONTH[m]` — indexing the 12-element const
`NTuple{12,Int}` lookup table by a **runtime** index. Minimal trigger:
`f(x::Int64) = (31,28,…)[x]` (and `MYTUP[x]` for a global const tuple).

Root cause: homogeneous-tuple dynamic indexing IS supported (calls.jl ~3114: materialize the
tuple's fields into a WasmGC array, `array.get` by `index-1`), but the gate at calls.jl:3107
only recognized the index when `field_ref isa Core.SSAValue`. When the index is a bare
function parameter — `getfield(tuple, _2, boundscheck)` where `_2` is a `Core.Argument` — it
fell to the `else → nothing` branch, skipped the dynamic handler, and stubbed to `unreachable`.
(Computed indices became SSAValues and worked; a directly-passed arg did not.)

Fix: accept `Core.Argument` alongside `Core.SSAValue` as a dynamic tuple index (one line).
Verified native==wasm via the bridge across the full index range and across element types —
Int (`(31,28,…)[x]`), Float (`(1.5,2.5,3.5)[x]`), Char (`('a','b','c')[x]`), and
arithmetic-derived args (`(10,20,30,40)[x+1]`). The array-build path is the same battle-tested
code the SSAValue case uses; only the guard was widened. loop_guard clean.
