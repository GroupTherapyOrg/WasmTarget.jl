## Analysis

**FIXED (P3 bugsmash).** Two narrow-int canonicalization bugs:

1. The ledger find: `zext_int(Int64, %1)` where `%1 = add_int(0x01, x)`
   :: UInt8. Sub-32-bit values live in i32 locals that carry dirty
   high bits (`0x01 + 0xff` leaves 0x100 — add_int does not
   re-narrow), and zext_int extended WITHOUT masking the source width,
   so the shift count became 256 instead of 0; the over-shift guard
   then produced 0 where native returns x. Fix: mask 8/16-bit sources
   (&0xFF / &0xFFFF) before I64_EXTEND_I32_U, in the i32→i32 "no-op"
   case, and in the 128-bit zext path (which previously didn't extend
   sub-32 sources at all).

2. Found probing the fix: SIGNED narrow shift results were
   zero-masked, not re-sign-extended — `Int8(-1) << 0` masked to 0xFF
   and read back as 255. _emit_shift_guarded! now takes signed_narrow
   and emits extend8_s/extend16_s instead of the mask for shl, and
   adds the same canonicalization after lshr (shift-0 of 0x80 must
   read back as Int8(-128), not 128).
