## Analysis
**Root-caused 2026-06-22 (wt-dyndispatch); DEFERRED (deep).** `Dict{Int64,Int64}(1=>10, 2=>20)`
resolves to the vararg ctor `Dict{K,V}(::Pair, ::Vararg{Pair})`. Base forwards it to the tuple
form `Dict{K,V}(ps)` (which IS overlaid → empty + setindex!, interpreter.jl:31), but the
closed-world pass leaves the vararg ctor an UNCOLLECTED `invoke` (pairs=1; the forward isn't
inlined/collected). WT then hits the fieldwise-constructor fallback at **invoke.jl:4709** — it
sees `is_struct_type(Dict{..})` and `struct.new`s the Dict from `(typeId, arg1, arg2)`, i.e. 3
operands for the Dict's ~9-field layout {typeId, slots, keys, vals, count, …} → `expected i64,
found ref` validation failure. The UNTYPED literal `Dict(1=>10,2=>20)` works because it routes
`dict_with_eltype → DT(K,V)(tuple) → the tuple overlay` directly (no uncollected vararg ctor).

Why not a quick fix:
- **Defensive guard idea** (sound, but doesn't close this): gate invoke.jl:4709 on
  `length(args) == length(field_types)` so the fieldwise `struct.new` fallback only fires for
  genuine positional constructors. For Dict (2 args vs ~9 fields) it would skip → fall through to
  a stub/trap. This is a real soundness improvement (the fallback currently mis-builds any
  arity-mismatched type, and could SILENTLY corrupt instead of validation-failing if arg/field
  wasm types coincided), but it only changes this gap's failure mode (validation→trap), not fix it.
- **Overlay the vararg ctor → tuple overlay** (tried, REVERTED): `@overlay Dict{K,V}(ps::Pair...) =
  Dict{K,V}(ps)` routes the typed literal to the tuple overlay, BUT it REGRESSES the working untyped
  path — adding the overlay prevents the untyped literal's const-fold, forcing the actual runtime
  Dict build, which THROWS. The runtime empty+setindex Dict build is itself FRAGILE: it works
  INLINED (`d=Dict{K,V}(); d[1]=10; d[2]=20` and even `for p in (1=>10,2=>20); d[p.first]=…`) and
  when const-folded, but the overlay's build compiled as a separate/non-folded function throws.
  So the deep blocker is the **fragile runtime Dict-build path** (+ vararg-ctor collection), not the
  fallback per se. Deep; deferred. See wt-soundness-loop memory "OVERNIGHT 2026-06-22".
