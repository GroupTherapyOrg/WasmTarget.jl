## Diagnostic update (P2-batch11, 2026-06-10)
Not a shift-guard bug. Under the overlay method table, inference resolves
`0x01 >> 0x8000000000000000` through a method whose body is the RAW
`lshr_int` intrinsic (raw semantics: huge amounts wrap -> returns 1), while
NATIVE dispatch selects the guarded `>>` method (returns 0). Confirmed:
`Base.lshr_int(0x01, 0x8000000000000000) == 1` natively while
`0x01 >> 0x8000000000000000 == 0`. The const-fold bakes the wrong branch;
the runtime path compiles the same wrong method body. ROOT CAUSE IS
METHOD SELECTION UNDER CC.OverlayMethodTable for mixed-width shifts (not
concrete-eval; reproduces with concrete_eval_eligible = :never too).

## Analysis
**Root cause: constant-folding divergence on an extreme shift literal.** Native Julia
folds `abs(Float64(0x01 << -9223372036854775808))` to the constant `0.0` (`@code_typed`
shows `return 0.0`) — `-9223372036854775808` is the Int64 typemin literal, and shifting
a `UInt8` by a magnitude ≥ its width is 0. WasmTarget's pipeline does **not** fold it
the same way: it emits the runtime shift and gets 1.

Note the common narrow-width shift bug is **fixed** ([[39f963bc044a]]): `0x01 << x` for
a runtime `x` is now correct across all edge amounts (the shift result is truncated to
the UInt8 width and the i64→i32 amount wrap saturates). This residual case is narrower:
it only manifests via the un-folded literal-typemin path that diverges from Base's
optimizer. **Deferred to Part 2** (inference/constant-folding parity with Base; the
runtime shift codegen itself is now correct for non-folded amounts).
