## Analysis
**Root cause: a *compiled* transcendental called with a CONSTANT arg traps as a map
kernel** — the map-kernel-dependency bug. Narrowed:
- `map(y->cos(0.5), v)` ✅, `map(y->sqrt(2.0), v)` ✅, `map(y->0.5, v)` ✅
- `map(y->atan(0.5), v)` ❌ traps, `map(y->asin(0.5), v)` ❌ traps
- `map(y->asin(y), v)` ✅ (variable arg — this is the already-fixed [[4c8ae5f6bb07]])

So `cos` (imported) and `sqrt` (a wasm instruction) work as map kernels, but `atan`
(a *compiled* Base function) traps **when its argument is a compile-time constant**.
`asin` is overlaid as `atan(x/√…)`, so `asin(1.0)`/`asin(0.5)` inherit the atan trap.
This is NOT a regression in the asin fix — the asin gap [[4c8ae5f6bb07]] (`asin(y)`,
variable) genuinely passes; this is a distinct, deeper manifestation of the
map-kernel-dependency class (cf. [[3b005c4957f7]], [[a771b83dda7c]]).

**Where to fix:** the codegen for a compiled function used as a map-kernel dependency
(constant-argument path). **Deferred to Part 2** (deep codegen).
