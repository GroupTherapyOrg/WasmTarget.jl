## Analysis
**ROOT-CAUSED ‚Üí TIER-1 (dynamic dispatch), 2026-06-17 loop.** Minimal trigger:
`Dict(Int32(0) => 0, 0 => 0)` ‚Üí **`Dict{Signed, Int64}`** (`Signed = typejoin(Int32,
Int64)`, ABSTRACT). The `sum`/`abs`/`cumsum` wrappers the shrinker keeps producing are
just disguises for "a second key of a different Int width", which widens the inferred key
type to the abstract `Signed`. Native returns `1` (both keys are `0`, equal); wasm traps.

This is the shared root of the whole ~11-gap "abstract-Dict-key (Int-widen)" cluster
(`run.jl rank` #1): `01e4e1bb61d3 05b2b084ef65 20775d3ec5b2 418e48316c5c 46a251368b26
a70595550bd8 b85535ae7df7 cb567b610585 e6c8c6edaeaa edf2c277a170 ef3c54645d9f`.

Why it's NOT a clean Tier-0 fix: an abstract-`Signed` key is boxed, and Base `Dict`'s
`ht_keyindex`/insert call `hash(key)` and `isequal(k1,k2)` on it ‚Äî which must DISPATCH to
the concrete type (`hash(::Int32)` vs `hash(::Int64)`). WT can't resolve that dynamic
call ‚Üí trap. So closing this cluster needs **Tier-1 dynamic dispatch** (the gated/perturbing
`WT_DYNDISPATCH` frontier), OR a narrow special-case (abstract-but-all-integer key ‚Üí
represent boxed key, unbox to i64 for a consistent hash/eq) ‚Äî intricate, deferred.
Re-tiered from T0‚ÜíT1; folded into the T1 dispatch frontier (task #33). See LOOP.md ¬ß1-2.

**DEEP DIVE 2026-06-22 (wt-dyndispatch branch) ‚Äî root pinned to TWO blockers; the
`BigInt ‚àà subtypes(Signed)` problem makes a *sound* runtime fix genuinely hard.**
Instrumented `collect_closed_world` per-variant (repros: /tmp/dict_variants.jl,
/tmp/r5_runtime.jl, /tmp/dump_func3.jl). Findings that CORRECT earlier notes:

1. **Construction root = `dict_with_eltype` is a dynamic `:call`, NOT an indexing trap.**
   For the untyped literal `Dict(Int32(0)=>0, 0=>0)` (what every cluster gap uses) the
   ONLY collected statement is `dynamic dict_with_eltype(DT, (0=>0,0=>0), %elt::Type)`.
   The 3rd arg stays abstract `::Type` because `eltype(ps)` is computed by a RUNTIME
   `promote_typejoin` loop over the tuple's Type parameters (types-as-values, flowing
   through œÜ nodes), which never folds (concrete_eval_eligible ‚áí `:none`). So the typed
   `dict_with_eltype` method can't be selected ‚Üí the whole call is stubbed. `which`-mapped:
   the literal resolves to `Dict(ps::Pair...)` = `Tuple{Type{Dict}, Vararg{Pair}}` (Base:115).

2. **A typejoin-preserving overlay routes it to a COLLECTABLE typed ctor ‚Äî and exposes
   the real wall.** Overlaying `(::Type{Dict})(ps::Pair...)` to splice `Dict{K,V}(ps)`
   (K,V via a `@generated` typejoin ‚Äî always const, the `_wt_type_name_str` trick) makes
   the construction collect cleanly (7 pairs incl `ht_keyindex2_shorthash!(Dict{Signed,Int64},
   Int32/Int64)`). It does NOT const-fold (verified: `length(Dict{Signed,Int64}(consts))`
   stays `invoke ctor; getfield :count` ‚Äî the earlier "r2 const-folds" belief was WRONG;
   r2 just left its ctor body *uncollected*). So the runtime build path MUST work.

3. **The killer: `BigInt <: Signed`.** With the body collected, the dynamic-dispatch
   discovery (`_dynamic_dispatch_candidate_mis`) fires on the abstract `hash(stored::Signed)`
   / `isequal(key, stored::Signed)` calls inside `ht_keyindex`. Its `isstructtype(Tp)` filter
   (trimcollect.jl:97) **REJECTS the actual Int32/Int64 primitives but ACCEPTS BigInt** (a
   struct subtype of Signed). Discovered candidates were literally `hash(BigInt,UInt64)`,
   `==(Int32,BigInt)`, `==(Int64,BigInt)`. The emitted dispatch ladder therefore has a bogus
   unsupported-BigInt branch and fails validation: `ref.test (ref $BigInt)` then narrow-store
   `local.set` into a structref local with NO `ref.cast` (`expected structref, found anyref`).

‚áí A sound runtime fix needs ALL of: (a) discovery to collect Int32/Int64 PRIMITIVE
candidates (relax isstructtype) AND **exclude BigInt/Int128/etc. that have no wasm rep**;
(b) `_try_inline_typeid_dispatch` to handle primitive candidates (cast boxed key ‚Üí numeric
box ‚Üí struct.get value ‚Üí call concrete hash/==); (c) fix the isa-narrow missing-`ref.cast`.
All three touch the WORKING struct-dispatch path (regression risk) for a LOW-PI cluster.
The one shortcut ‚Äî widen integer keys to a single concrete type (`Dict{Int64,Int64}`) ‚Äî is
VALUE-correct for these gaps' ops (length/isempty/haskey/get; integer hash/eq are width-
agnostic) but TYPE-divergent (`keytype(d)`/`typeof(first(keys(d)))` would differ Signed‚ÜíInt64),
so it violates the strict soundness mandate. **Decision: still deferred ‚Äî deeper than an
overlay, and not soundly closable without the full primitive-dispatch + BigInt-exclusion build.**
The typejoin overlay (sound, reverted ‚Äî closes 0 gaps alone) is the right groundwork when this
is picked up.
