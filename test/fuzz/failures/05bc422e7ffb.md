## Analysis
**Root cause: `uppercase`/`lowercase` only accept `String`, not `SubString`.** They
are handled by the `str_uppercase`/`str_lowercase` codegen intrinsic, which dispatches
on `String`; `chomp(s)` returns a `SubString{String}`, so `uppercase(chomp(""))` hits
no method and raises a compile error.

**Why not overlay-fixable yet (a trialed overlay was reverted).** The natural overlay
— a byte-loop reading `codeunit(s,i)` from the SubString into a fresh `String` —
*compiles* but is **silently wrong**: in that nested-build context `codeunit(::SubString)`
reads return 0, so `length(uppercase(chomp("ab\n")))` is right (2) while the bytes come
out zero (verified: `codeunit(uppercase(chomp("ab\n")),1)` = 0, want 65). A loud
compile error beats silently-wrong content. Same SubString/`String(bytes)` codegen
class as the strip gaps ([[0beb5ec969a2]]).

**Where to fix:** make `str_uppercase`/`str_lowercase` accept `AbstractString`, AND fix
the underlying `codeunit(::SubString)`-in-nested-build bug. **Deferred to Part 2.**
