## Analysis
**FIXED** — `reduce`/`foldl` over a `Vector` were unimplemented.

`reduce(min, sort(v))` lowers (via `mapreduce`) to an inlined `slt_int`+`ifelse`
loop for the runtime-length path, **plus** a `Base.mapreduce_impl(identity, min,
v, 1, n, 1024)` invoke for the large-vector (>1024-block) branch. That
`mapreduce_impl` block emits invalid wasm, so the whole module fails to validate
**even for a 4-element vector that never executes that branch** — every
`reduce`/`foldl` trapped. In lax (non-strict) mode the stub returned garbage: this
gap recorded `wasm=11` = `3 + 8`, i.e. `reduce(min, …)` yielded the **max** (8),
not the min (1).

Fix (`interpreter.jl`): overlay `Base.reduce(op::F, ::Vector)` and `Base.foldl`
with a plain left-fold (`_wasm_reduce_loop`), exact for the generated associative
ops (+, *, min, max). The overlay replaces the entire native lowering, so the
invalid `mapreduce_impl` block is gone. `op::F` forces per-op specialization so
the empty-collection identity (`zero`/`one` for +/*, throw for min/max) folds to a
compile-time constant — otherwise `op` infers as abstract `Function` and the empty
branch becomes a `dynamic invoke Base.reduce_empty(…)::Union{}` that won't compile.
Regression-guarded by `@testset "reduce/foldl(min/max)"` in runtests.jl.
