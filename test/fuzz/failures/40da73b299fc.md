## Triage (P3, 2026-06-12)

Two distinct catch-arm bugs reproduce from this construct standalone
(`nt(x) = try (try div(x,x) catch; 0x00 << length([0,0,0]) end) catch; 0x00 end`):

1. **x=0 (catch path): uncatchable "array element access out of bounds".**
   The catch arm builds `[0,0,0]` via the tuple fill loop:
   `%9 = φ(#5 => 1, #10 => %19)` — the φ INIT edge originates at the
   catch-entry block #5 (no terminator, falls through into the loop
   header #6). In catch-arm subset emission that fall-through φ store is
   never emitted, the index local defaults to 0, and
   `getfield((0,0,0), 0, false)` → array[-1] → OOB trap.
   Same fill loop compiles correctly in main-flow context.

2. **x=5 (no-throw path): escaped WebAssembly.Exception.**
   `%37 = φ(#4 => %4::Int64, #12 => %33::UInt8)::Union{Int64,UInt8}` —
   a small-union φ. The π(%37, Int64) extraction reads the union local
   wrong (tagged-union unbox), so the UInt8 trunc round-trip check
   `eq_int(%40, zext(trunc(%40)))` fails for value 1 and the
   throw_inexacterror fires OUTSIDE any try region (IR #17 is after
   leave/pop_exception) — the tag-0 throw escapes to the host.
   Native never reaches #17.

Fix both in the stackified/catch-arm emitters; likely also the root of
gaps 600287f76223 / 464d3b1b41ec / 0d6aa24022dd / ae64a1ba676e
(catch-arm φ routing family).
