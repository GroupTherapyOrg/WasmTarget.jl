## Analysis
**Root cause: non-ASCII `Char` case conversion needs Unicode tables (feature).**
`uppercase('é')` → 'É' (codepoint 201) requires Unicode simple-case-folding tables.
ASCII works (`uppercase('a')` → 'A', verified ✅); the non-ASCII path traps.

This is the **`Char`-level** analogue of the ASCII-only `str_uppercase`/`str_lowercase`
string intrinsics — char case folding for codepoints ≥ 0x80 is unimplemented, so the
dispatch falls through to a Base path that traps in wasm. Shares its root with
[[bc93362af34e]] (`isspace('é')`) and [[6bd3c602cf17]] (`isdigit(uppercase('é'))`):
all need Unicode character-property / case tables.

**Where to fix:** add WasmGC-friendly Unicode case + category tables (or a compact
range-encoded form) and wire `uppercase`/`lowercase`/`isspace`/`isdigit`/… on `Char`
to them. **Deferred to Part 2** (feature — Unicode tables, not overlay-fixable).
