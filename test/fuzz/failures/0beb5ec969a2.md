## Analysis
**`strip`/`lstrip`/`rstrip` on multibyte UTF-8.** Diagnostic: `length(strip("héllo"))`
gives wasm=4 vs native=5 — strip drops a byte of the multibyte `é`, leaving a
truncated string whose char count is off.

Root cause is two **underlying codegen bugs** the current strip overlays contort
around (they byte-index with `length`, the char count, which only happens to work
for pure-ASCII):

1. **`ncodeunits` on a `String(bytes)` result returns the wrong value** (aliasing —
   the freshly-built String's length field isn't read correctly). A correct strip
   must rebuild the trimmed string and re-measure it, which trips this.
2. **A complex boolean condition + `break` inside a `while`** miscompiles, so the
   single-pass whitespace-trim loop stops trimming early.

A correct `ncodeunits`-based single-pass strip needs (1) and (2) fixed first; then
strip becomes trivial. Note `length(::String)`, `startswith/endswith`, and
`reverse(::String)` unicode paths are now all fixed — strip is the last
unicode-string holdout. **Deferred to Part 2** (deep codegen, not overlay-fixable
without those two fixes). See also `FINDINGS.md`.
